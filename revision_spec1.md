## 1) 최우선: “데모 앱”을 “Presales 산출물”로 바꿔라

현재 구현 목록만 보면 훌륭하다.

하지만 평가자가 보는 건 구현 목록이 아니다.

그들은 결국 이렇게 판단한다.

- 이 사람이 그냥 데이터를 붙였나?
- 아니면 고객 문제를 PoC 제안으로 구조화할 수 있나?

그래서 다음 단계의 핵심은 앱 자체가 아니라 **앱 바깥의 산출물**이다.

### 바로 필요한 것

1. **대표 시나리오 1개 고정**
    - Lung Cancer / KRAS / Literature Intelligence 같은 식으로 하나를 메인 시나리오로 확정
    - 나머지 preset은 보조로만 둔다
2. **이 시나리오 기준으로 5개 화면 캡처 세트 확보**
    - Executive Overview
    - Evidence Explorer
    - Trial Landscape
    - PoC Designer
    - Reference Architecture
3. **각 화면 아래 한 줄 해석 문장 고정**
    - “무엇을 보여주는가”
    - “왜 고객에게 중요한가”
    - “어떤 PoC로 이어지는가”

이 작업이 먼저다.

왜냐하면 포트폴리오에서 앱은 “보여주는 물건”이지 “읽게 만드는 물건”이 아니기 때문이다.

---

## 2) 다음 우선순위: “신뢰성 표시 레이어” 추가

지금 기능은 충분하지만, 포트폴리오 관점에서 가장 중요한 건 **신뢰성 장치가 눈에 보이느냐**다.

아래 5개는 거의 필수다.

### A. 데이터 출처 배지

각 화면 상단 또는 카드에 명시

- Source: PubMed
- Source: ClinicalTrials.gov
- Generated by rules-based PoC framing
- Fallback data used 여부 표시

### B. 생성 시각 / 조회 시각

- “Last fetched at …”
- 평가자 입장에서 “실시간/재현 가능” 느낌을 준다

### C. Fallback 사용 여부 배지

- API 실패 시 fallback이면 반드시 표시
- 이걸 숨기면 오히려 불신을 산다

### D. Non-diagnostic disclaimer 고정

- 의료 판단 아님
- 연구/제안용 데모
- 이 문구는 더 눈에 띄게

### E. Query summary bar

모든 화면 상단에 고정

- Disease
- Target
- Objective
- Region
- Time Window

이 5개가 있어야 앱이 더 “업무용 콘솔”처럼 보인다.

---

## 3) 그다음: “Export 가능한 제안서 한 장”을 만들어라

이게 다음 레버리지 포인트다.

지금 앱은 화면상으로는 좋다.

하지만 Presales 포지션이라면 평가자는 무의식적으로 이렇게 본다.

> “그래서 이걸 고객 미팅 이후 어떤 문서로 넘길 수 있지?”
> 

따라서 지금 가장 ROI 높은 기능 추가는 챗봇이 아니라 **Export Summary**다.

### 추천 기능

버튼 하나:

**Generate PoC Brief**

출력 형식:

- Query summary
- Opportunity statement
- Top evidence themes
- Trial signals
- Recommended PoC type
- 6-week scope
- KPI
- Risks / Guardrails
- Reference architecture summary

형태는 처음엔 PDF가 아니어도 된다.

- Markdown
- Printable HTML
- Copyable text block

중요한 건 **“앱 → 제안서 초안”** 흐름이 보이는 것이다.

이 기능 하나가 들어가면 앱 성격이 달라진다.

그냥 대시보드가 아니라 **Presales workbench**가 된다.

## 4) 그다음: 검증 매트릭스를 만들어라

지금 구현됐다고 끝이 아니다.

포트폴리오에서 더 중요한 건 **일관되게 재현되느냐**다.

### 해야 할 검증

대표 preset 3개 × objective 3개 정도로 테스트 표를 만든다.

예시 컬럼:

- Query
- Objective
- Executive output 정상 여부
- Evidence results 정상 여부
- Trial results 정상 여부
- PoC type 적절성
- KPI 적절성
- Fallback 발생 여부
- Notes

이유는 단순하다.

면접에서 “어떻게 검증했나요?”라는 질문이 나오면

“돌려봤습니다”는 약하고,

“preset-query matrix로 outcome consistency를 점검했습니다”는 강하다.

## 3) 세 번째 위험: 라이브 데모에 과신

포트폴리오/면접에서는 의외로 네트워크, API 속도, 배포 이슈가 자주 발목을 잡는다.

즉, 배포는 필요하지만 **신뢰성 고정 후**가 맞다.
