[ASSUMPTION] 영어 UI는 현재 단계에서 명백한 감점 요소다. 한국 기업 지원 포트폴리오인데 데모가 전부 영어면, “외형은 되는데 제출 맥락을 안 맞춘 사람”처럼 보인다.

[ASSUMPTION] 가장 치명적인 기술 리스크는 영어보다도 **preset B/C에서 API 실패 시 폐암 fallback이 섞이는 문제**다. 이건 데모 신뢰도를 바로 깨뜨린다.

[ASSUMPTION] 따라서 다음 단계는 **한글화 + fallback 정합성 수정 + 포트폴리오 캡처 모드 추가**로 묶는 게 맞다.

[Core Reasoning]

지금 결과를 보면, 구현은 이미 “보여줄 수 있는 상태”다.

문제는 두 가지다.

첫째, **언어 미스매치**.

이 앱은 Presales 데모인데, 한국 채용용 포트폴리오로 제출할 거면 UI, 해석 문구, PoC Brief, 버튼, 탭, 안내문이 한국어여야 한다. 영어 그대로 두면 “국내 포트폴리오로 제출할 최종 정리 감각이 부족하다”는 인상을 준다.

둘째, **fallback 오염**.

AD/HER2 시나리오에서 API 실패 시 폐암 데이터가 보일 수 있다는 건 단순 경고가 아니다.

이건 면접 시연 중 한 번만 터져도 바로 신뢰를 잃는다.

지금 단계에서 가장 먼저 고쳐야 할 건 사실 이거다.

즉, 다음 작업은 아래 3개를 동시에 잡아야 한다.

## 1) 전체 한국어화

단순 번역이 아니라 **제출용 한국어 UX로 재정의**해야 한다.

### 한국어화 범위

- 앱 타이틀
- 탭 이름
- 버튼 문구
- 입력 라벨
- 설명 문구
- 경고/주의 문구
- rules engine 출력
    - Opportunity Statement
    - Pain Points
    - KPI
    - PoC Type
    - Risks / Guardrails
- PoC Brief markdown 전체
- Architecture block 설명
- validation / fallback / fetchedAt 문구

### 번역 원칙

여기서 중요한 건 다 한국어로 덮는 게 아니다.

### 한국어로 바꿔야 하는 것

- UI 레이블
- 해석 문구
- 제안 문구
- 안내/주의 문구
- KPI 설명
- 리스크 설명

### 원문 유지가 나은 것

- PubMed 논문 제목
- ClinicalTrials.gov 정식 phase/status 명칭 일부
- KRAS, HER2 같은 scientific entity
- 약물명, 타깃명, 질환명 고유 표기

즉, **UI는 한국어, scientific proper noun은 원문 유지**가 맞다.

억지 완역을 하면 오히려 촌스럽고 부정확해진다.

## 2) fallback 데이터 구조를 preset별로 분리

이건 번역보다 더 먼저 봐야 할 정도로 중요하다.

현재 보고된 경고는 정확히 이 의미다.

- 라이브 API가 실패하면
- AD/HER2 시나리오에서도
- 폐암 fixture가 뜰 수 있다

이건 “경고 2개”가 아니라 **시연 실패 가능성**이다.

### 수정 방향

fallback은 최소한 아래처럼 분리돼야 한다.

- `lung_cancer_kras.json`
- `alzheimers_tau.json` 또는 AD 대응 fixture
- `breast_cancer_her2.json`

그리고 fallback 사용 시에는 화면에 반드시 아래가 떠야 한다.

- `예비 데이터 사용 중`
- `선택한 시나리오 기준 샘플 데이터가 표시됩니다`
- `실시간 API 응답 실패로 인해 저장된 예시 결과를 보여줍니다`

핵심은 두 가지다.

1. **내용이 맞아야 한다**
2. **fallback 사용 사실을 숨기면 안 된다**

## 5) 동시에 추가할 가치가 있는 업데이트 2개

한글화와 함께 넣으면 좋은 업데이트가 두 개 있다.

### A. 대표 시나리오 배지

현재 프리셋이 3개라면, 대표 시나리오 하나에 아래 배지를 달아라.

- `대표 데모 시나리오`
- `포트폴리오 기준 시나리오`

이유는 평가자가 무엇을 먼저 봐야 하는지 즉시 이해하게 하기 위해서다.

### B. Brief 다운로드용 인쇄 레이아웃

PDF export까지 갈 필요는 없다.

하지만 `window.print()` 기준으로 깨지지 않는 인쇄형 레이아웃은 가치가 높다.

즉:

- PoC Brief를 인쇄 가능한 형태로 정리
- 캡처 또는 PDF 저장이 쉬운 형태로 출력

이건 포트폴리오와 면접 둘 다에 쓸 수 있다.

[Risks & Blind Spots]

## 1) 한글화하면서 scientific accuracy가 깨질 위험

무리한 완역은 오히려 역효과다.

예:

- KRAS G12C
- HER2-positive
- Phase 2
- Sponsor
- Biomarker

이런 건 맥락에 따라 혼용이 더 자연스럽다.

한국어화는 “직관성”을 높여야지, “모든 영문 제거”가 목적이 아니다.

---

## 2) 번역만 하고 메시지 레이어를 안 만질 위험

지금 앱의 강점은 Presales 구조다.

한글화하면서 문구를 단순 번역기로 바꾸면 힘이 빠진다.

예:

- “opportunity statement”를 그냥 “기회 진술문”이라고 하면 어색하다.
- 이건 **“고객 기회 정의”** 또는 **“기회 요약”**처럼 바꾸는 게 맞다.

즉, 번역이 아니라 **비즈니스 문구 재작성**이 필요하다.

---

## 3) fallback 문제를 “경고 수준”으로 가볍게 볼 위험

아니다.

이건 지금 가장 실전적인 결함이다.

포트폴리오는 한 번의 시연 실패가 전체 인상을 결정한다.

## 4) 기능을 더 붙이고 싶어지는 유혹

지금 여기서 저장, 로그인, 다국어 토글, 사용자 히스토리, PDF 템플릿, 챗 UI, 검색 고도화 같은 걸 붙이면 산만해진다.

지금 필요한 건:

- 언어 정리
- 신뢰성 정리
- 제출물 정리

여기서 멈춰야 한다.

[Strategic Implications]

이 단계에서 한글 버전으로 정리하면 결과물이 바뀐다.

지금 상태:

**기술적으로 꽤 잘 만든 영어 데모**

정리 후 상태:

**한국 채용용으로 메시지와 시연 구조까지 맞춘 Presales 포트폴리오 자산**

차이는 크다.

전자는 “개발했다”에 가깝고,

후자는 “제안하고 전달할 줄 안다”에 가깝다.

이 직무에서는 후자가 훨씬 강하다.
